# [SD] 운영 메뉴얼 (실행 순서)

이 문서는 `shopee-dashboard`를 실제 운영할 때, 상품 수집부터 멀티플랫폼 리스팅/가격수정까지를 빠르게 수행하기 위한 실무용 메뉴얼이다.

## 1) 오늘 작업 시작 전 체크

먼저 토큰 상태와 시스템 상태를 확인한다.  
`[sd]` 토큰 헬스 알림에 치명 에러가 없고, 플랫폼 API 호출이 가능한 상태인지 확인한 뒤 시작한다.

그다음 오늘 우선순위 상품군을 정한다.  
신보, 가격 변동이 큰 상품, 재고 회전이 빠른 상품을 먼저 처리한다.

## 2) 원천 데이터 수집

등록할 상품의 원천 데이터를 모은다.  
필수 항목은 상품명, 아티스트/그룹, 발매일, 버전/옵션, 매입가, 통화, 이미지, 무게/크기다.

수집 시 주의할 점은 같은 앨범의 일반판/특전판/세트판을 섞지 않는 것이다.  
버전이 다르면 SKU를 분리한다.

## 3) Master Data 저장

수집한 데이터는 플랫폼 입력 전에 먼저 Master Data에 저장한다.  
핵심은 “플랫폼 등록”이 아니라 “정확한 기준 데이터”를 먼저 고정하는 것이다.

저장 시에는 내부 SKU, 원천 출처, 수집 시각, 가격 근거를 함께 남긴다.  
이력이 남아야 나중에 가격 롤백과 원인 추적이 가능하다.

## 4) 등록 전 검증 (필수)

아래 항목을 통과하지 못하면 등록하지 않는다.

- 필수 필드 누락 없음 (상품명/카테고리/가격/이미지/무게 등)
- 중복 SKU 없음
- 통화/가격 단위 정상
- 이미지 URL 유효
- `condition = NEW` 등 preset 기본값 정상

K-POP 신보는 속도가 중요하지만, 오등록 1건이 손실을 더 크게 만든다.  
검증은 생략하지 않는다.

## 5) 플랫폼별 Dry-Run

Master Data를 각 플랫폼 payload로 변환한 뒤 Dry-Run을 먼저 실행한다.  
Shopee/Joom/Qoo10/eBay/Alibaba 중 오늘 운영 대상 플랫폼만 선택해서 진행한다.

Dry-Run에서 실패한 항목은 원인별로 분류한다.
- 데이터 오류
- 매핑 오류
- 플랫폼 제약/권한 오류

실패 건은 수정 후 Dry-Run을 재실행하고, 통과한 건만 실제 반영한다.

## 6) 실제 리스팅/수정 실행

실행은 batch 단위로 진행한다.  
한 번에 전체를 밀지 말고, 위험도 높은 상품군은 작은 배치로 나눠 처리한다.

실행 후에는 성공/실패, 플랫폼 item id, request id, 재시도 여부를 기록한다.  
실패 건은 “즉시 재시도 가능”과 “수동 확인 필요”로 분리한다.

## 7) 가격 수정 운영 규칙

가격은 매입가/환율/수수료 변화에 따라 업데이트한다.  
변경 전후 값은 반드시 snapshot으로 남긴다.

권장 운영:
- 소폭 변동: 로그만 남기고 보류 가능
- 중간 변동: 자동 반영 가능
- 큰 변동/마진 위험: 승인 후 반영
- 비정상 값(음수 마진 등): 차단

실행 전에는 항상 Dry-Run diff를 보고, 실행 후엔 결과를 batch 단위로 점검한다.

## 8) 롤백 기준

가격/리스팅 변경이 잘못 반영되면 즉시 롤백한다.  
롤백은 “이전 정상값(last known good)” 복원을 기준으로 한다.

롤백 시에는 사유, 작업자, 대상 batch를 기록한다.  
같은 오류가 반복되면 룰/매핑을 수정한다.

## 9) 일일 마감 체크

하루 작업 종료 전 아래를 확인한다.

- 실패 배치 잔여 건수
- 승인 대기 건수
- 가격 위험 건수(마진 임계치)
- 플랫폼별 동기화 불일치(drift)
- 다음날 우선 처리 상품군

## 10) 주간 운영 루틴

주 1회는 전략 관점에서 점검한다.

- source 품질/신선도 추세
- 반복 실패 원인 Top 3
- preset/매핑 업데이트 필요 항목
- 플랫폼 확장 우선순위 재조정

핵심 원칙은 동일하다.  
**Master Data를 먼저 정확하게 만들고, 그 데이터를 기준으로 각 플랫폼에 안전하게 배포한다.**
